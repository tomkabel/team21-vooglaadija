The Architecture

The Birth & The Bridge

When Job #4473 is born, it is written to the database twice — once as a job record, once as an outbox event — inside a single transaction.

If the API crashes one millisecond after commit, both records survive. If it crashes before commit, both disappear.

There is no partial state.

  • Job record + outbox event are always in sync
  • Idempotent: duplicate outbox entries are silently skipped
  • No compensating transactions needed
job = DownloadJob(
    id=job_id,
    user_id=current_user.id,
    url=data.url,
    status="pending"
)
db.add(job)
await write_job_to_outbox(db, job_id)
# Same session, same transaction
await db.commit()
# Atomic: both or neither

The Architecture

The Outbox Relay: A Bridge, Not a Warehouse

Every 30 seconds, the outbox relay queries PostgreSQL for pending entries — skipping anything another relay instance is already handling. After publishing to Redis, entries are deleted. The table stays empty. No bloat. No archive. 

async def sync_outbox_to_queue(batch_size: int = 100) -> int:
    claim_result = await db.execute(
        select(Outbox)
        .where(Outbox.status == "pending")
        .order_by(Outbox.created_at)
        .limit(batch_size)
        .with_for_update(skip_locked=True)
    )
    entries = claim_result.scalars().all()

    for entry in entries:
        await redis_client.lpush("download_queue", str(entry.job_id))
        await db.execute(delete(Outbox).where(Outbox.id == entry.id))
    await db.commit()

The Architecture

Exactly-Once Delivery Is a Myth

Redis can deliver the same job to two workers simultaneously. So workers do not ask permission — they claim the job with a guarded atomic update that only succeeds if status is still pending. 

result = await db.execute(
    update(DownloadJob)
    .where(
        DownloadJob.id == job_id,
        DownloadJob.status == "pending"
    )
    .values(status="processing", updated_at=datetime.now(UTC))
)
await db.commit()

claimed = result.rowcount == 1
if not claimed:
    return False  # Expected — not an error

At-least-once delivery + idempotent state transitions = effective exactly-once processing.

The Storm

SIGKILL & SIGTERM: Two Failures, One Recovery

SIGKILL — When Graceful Shutdown Is Impossible

We kill the worker — not gracefully. SIGKILL, like an OOM kill in Kubernetes. The shutdown handler never runs. Job #4473 is stranded in processing state.

Fifteen minutes later, the zombie sweeper finds it, marks it pending, and another worker picks it up.

Critical fix: recovery now routes through the transactional outbox — not a naked dual-write.

SIGTERM — The Polite Failure

For planned shutdowns, we handle SIGTERM with a 25-second grace period.

  • t=0 — SIGTERM received; shutdown_event.set()
  • t=0–25 — Current job completes OR is requeued via outbox
  • t=25 — Worker exits cleanly
  • t=30 — Kubernetes sends SIGKILL (5s runway)
