Scene 1 — 0:45

What Can Go Wrong

The job will face rate limits, crashed workers, and killed processes. It still completes.

HTTP 429

YouTube rate-limits aggressively. The system backs off with jitter.

SIGKILL

Worker killed mid-download. Zombie sweeper requeues the job.

Recovered

File completes. Metrics confirm. Logs prove it.

Scene 3 — Beat A — 2:00

The Birth: Atomic Outbox Write

The job is written to the database twice — inside a single transaction.

Code
# app/api/routes/downloads.py
async def create_download(...):
    job = DownloadJob(url=url, status="pending")
    db.add(job)

    outbox = OutboxEntry(
        aggregate_id=job.id,
        event_type="job_created"
    )
    db.add(outbox)

    await db.commit()

Scene 3 — Beat B — 2:45

The Bridge: Outbox Relay

Polls every 30 seconds. Scales horizontally with no extra coordination.

SQL
-- worker/processor.py
SELECT id, event_type, payload
FROM outbox
WHERE processed_at IS NULL
ORDER BY created_at
FOR UPDATE SKIP LOCKED
LIMIT 10;
Relay Workers
Relay 1
claims row A
[ LOCK ]
Relay 2
skips row A
SKIP LOCKED
Relay 3
claims row B
[ LOCK ]

After publishing to Redis, the outbox entry is deleted. The table stays empty. The outbox is a bridge, not a warehouse.

Scene 3 — Beat C — 3:45

The Claim: At-Least-Once Delivery

Exactly-once is impossible with Redis. The claim mechanism makes duplicates harmless.

Code
# worker/processor.py
async def claim_job(job_id: UUID):
    result = await db.execute(
        update(DownloadJob)
        .where(
            DownloadJob.id == job_id,
            DownloadJob.status == "pending"
        )
        .values(status="processing")
    )

    return result.rowcount() == 1

Zombie Sweeper
# worker/zombie_sweeper.py
async def requeue_stuck_jobs(timeout_minutes: 15):
    await db.execute(
        update(DownloadJob)
        .where(
            DownloadJob.status == "processing",
            DownloadJob.updated_at < now() - interval("15 minutes")
        )
        .values(status="pending")
    )
Scene 4 — Beat C — 6:30

Graceful Shutdown

SIGTERM gives the worker 25 seconds to finish or requeue. SIGKILL is the last resort.

→
t = 1s
Stop Polling
→

SIGKILL
25-second app timeout. 5-second buffer before SIGKILL.

All work is accounted for: completed, requeued, or swept.

A project without acknowledged boundaries is incomplete.

Scene 6 — 9:40
Gaps: RS256, bulkhead pattern, LISTEN/NOTIFY.

These are documented boundaries, not oversights.

Not Built

RS256 key rotation adds complexity unnecessary for a single-issuer course project.

Not Built

Bulkhead pattern would isolate worker pools. Out of scope for current scale.

Not Built

LISTEN/NOTIFY would reduce outbox latency from 30s to sub-second. Sacrificed for simplicity.
