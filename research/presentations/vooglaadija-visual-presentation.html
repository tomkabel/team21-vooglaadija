Outbox Pattern: Overview

The foundation of crash-proof job processing

API receives download request
↓
SINGLE PostgreSQL Transaction
INSERT job + INSERT outbox_entry
↓
COMMIT = both succeed | ROLLBACK = both fail
↓
Outbox Relay polls every 30s → publishes to Redis
"If the API crashes after the commit, both records survive. Job is never lost — delayed by at most 30 seconds."

Outbox Pattern: Recovery Scenario

# CRASH RECOVERY TIMELINE t=0ms API receives request t=5ms PostgreSQL COMMIT (job + outbox committed) t=6ms API process crashes ⚠️ t=30030ms Outbox relay polls, finds pending entry t=30035ms Relay publishes to Redis t=30040ms Job is processing # RESULT: Job is processed (30 second delay is acceptable)

✓ Atomic

Job and outbox in same transaction

✓ Crash-Safe

Committed data survives process crash

✓ Scalable

FOR UPDATE SKIP LOCKED enables horizontal scaling

Atomic Job Claims

No locks, no coordination, no distributed consensus needed

# Worker claims job atomically with UPDATE WHERE clause UPDATE download_jobs SET status = 'processing', updated_at = now() WHERE id = $job_id AND status = 'pending' -- WHERE clause is the key RETURNING rowcount IF rowcount = 1 → Job claimed ✓ IF rowcount = 0 → Already claimed by another worker → skip

No SELECT FOR UPDATE

Race handled by UPDATE WHERE clause

No Pessimistic Locking

Reduced contention

No Distributed Locks

No Redis/ZooKeeper needed

MVCC Guarantees

Database handles atomicity

Graceful Shutdown

30-second grace period: finish current job OR atomically requeue it

SIGTERM RECEIVED at t=0
↓
PHASE 1: readiness_probe → FAIL
Kubernetes/Load Balancer routes traffic elsewhere
↓
PHASE 2: 30-second grace period
Complete current job OR requeue atomically + cleanup partial files
↓
PHASE 3: Exit cleanly (or SIGKILL at t=60)
"This is NOT 'zero lost work' — it is 'all work is accounted for.'"

Exponential Backoff with Jitter

delay = min(base × 2^attempt, max_delay) + uniform(0, base)

Example: base=60s, max=600s

60-120s
Attempt 1
120-180s
Attempt 2
240-300s
Attempt 3
480-540s
Attempt 4
FAIL
Attempt 5 (max)

Without Jitter

100 jobs fail at t=0

→ All 100 retry at t=120 → THUNDERING HERD

With Jitter

100 jobs fail at t=0

→ ~20 retry at t=120, ~20 at t=130... → Smooth distribution

Code: Outbox Transaction

app/services/outbox_service.py::create_job_with_outbox (lines 25-45)

# Create job and outbox in SINGLE TRANSACTION async def create_job_with_outbox( db: AsyncSession, user_id: UUID, url: str ) -> DownloadJob: job = DownloadJob( id=UUID(), user_id=user_id, url=url, status=JobStatus.PENDING, created_at=datetime.now(UTC) ) outbox_entry = Outbox( id=UUID(), job_id=job.id, event_type="job_created", status=OutboxStatus.PENDING, created_at=datetime.now(UTC) ) db.add(job) db.add(outbox_entry) await db.commit() # ATOMIC: both succeed or both fail return job

Key: Line 38 — single db.commit() makes this atomic. If we crash after commit, both job and outbox survive.

Code: Outbox Relay

worker/outbox_relay.py::poll_and_publish (lines 45-80)

async def poll_and_publish(self, batch_size: int = 100): async with self.db.transaction(): entries = await self.db.execute( select(Outbox) .where(Outbox.status == OutboxStatus.PENDING) .order_by(Outbox.created_at) .limit(batch_size) .with_for_update(skip_locked=True) # KEY: Scale-safe ) for entry in entries.scalars(): await self.redis.lpush("download_queue", str(entry.job_id)) await self.db.execute( update(Outbox) .where(Outbox.id == entry.id) .values(status=OutboxStatus.ENQUEUED) )

Key: with_for_update(skip_locked=True) — Multiple relays can run simultaneously without deadlocks.

Code: Atomic Job Claim

worker/processor.py::claim_job (lines 30-55)

# No locks needed - UPDATE WHERE clause handles race async def claim_job(self, job_id: UUID) -> bool: result = await self.db.execute( update(DownloadJob) .where( DownloadJob.id == job_id, DownloadJob.status == JobStatus.PENDING # WHERE clause is key ) .values( status=JobStatus.PROCESSING, updated_at=datetime.now(UTC) ) ) claimed = result.rowcount == 1 if not claimed: # Another worker got it — expected, not an error return False return True

Key: The WHERE status=pending in the UPDATE is the magic. Only one worker succeeds.

async def _signal_handler(self, sig: signal.Signals): self.shutdown_event.set() self.logger.warning("SIGTERM received, initiating graceful shutdown") # Stop accepting new work (for Kubernetes) self.readiness_probe.set_unhealthy() # Wait for current job with timeout if self.current_job_id: try: await asyncio.wait_for( self._wait_for_job_completion(), timeout=30.0 # 30-second grace period ) except asyncio.TimeoutError: # Grace period exceeded — requeue atomically await self._requeue_job(self.current_job_id) self.logger.warning(f"Job {self.current_job_id} requeued after graceful shutdown")

Failure Scenarios

Understanding failure modes — the senior engineering difference

🔴 Scenario: Redis is Down

What Happens:
  • API can still accept downloads (writes to PostgreSQL + outbox)
  • Outbox relay cannot publish to Redis
  • Outbox entries accumulate
  • When Redis recovers, relay publishes accumulated entries
Result:
Jobs delayed but NOT lost
Recovery is automatic

Failure Scenarios (cont.)

🔴 PostgreSQL is Down

  • API returns 503 Service Unavailable
  • Health check fails
  • Kubernetes routes traffic to other pods
  • Jobs in Redis queue remain (worker can't acknowledge)
  • When PostgreSQL recovers, worker picks up where it left off
System degrades gracefully. No data loss.

tests/test_postgres_failure.py

🔴 Outbox Relay Crashes

  • Relay crashes mid-processing
  • Entries remain in 'pending' or partially processed state
  • Next relay instance picks up with FOR UPDATE SKIP LOCKED
  • Idempotency prevents duplicate publishing
No duplicate jobs. Recovery is automatic.
