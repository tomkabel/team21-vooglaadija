# Plan: Docker Compose Integration Test Workflow

## Overview
Create a GitHub Actions workflow that tests docker-compose builds and verifies all 5 services are healthy by checking health endpoints and logs.

## Services (5 total)
| Service | Health Method | Port | Notes |
|---------|--------------|------|-------|
| api | HTTP health endpoint | 8000 | Has Docker healthcheck |
| worker | Log pattern matching | - | No HTTP healthcheck, check logs |
| db | pg_isready | 5432 | Has Docker healthcheck |
| redis | redis-cli ping | 6379 | Has Docker healthcheck |
| nginx | HTTP via proxy | 8080 | Proxies to API:8000 |

## Workflow Triggers
- push: [main, develop]
- pull_request: [main, develop]
- workflow_dispatch

## Job Steps

### 1. Setup
- Checkout repository
- Set up Docker Buildx
- Verify nginx config exists (./infra/nginx/default.conf)

### 2. Build & Start Services
```bash
docker-compose up -d --build
```

### 3. Wait for Health Checks (Docker-level)
Poll each service with healthcheck until healthy:
- api: `docker inspect --format='{{.State.Health.Status}}'` → "healthy"
- db: `docker inspect --format='{{.State.Health.Status}}'` → "healthy"  
- redis: `docker inspect --format='{{.State.Health.Status}}'` → "healthy"

Timeout: 120 seconds
Retry interval: 5 seconds

### 4. Check Worker Health (Log-based)
Since worker has no HTTP endpoint, verify by checking logs:
```bash
docker-compose logs worker 2>&1 | head -20
```
Look for patterns:
- "Worker started"
- "Connected to Redis"
- "Waiting for jobs"
- No ERROR/FATAL messages

### 5. Verify Health Endpoints
Test both paths:
```bash
# Direct API health check
curl -s http://localhost:8000/api/v1/health | grep -q "ok"

# Via nginx proxy
curl -s http://localhost:8080/api/v1/health | grep -q "ok"
```

### 6. Verify Service Logs (Post-health)
Check recent logs to confirm healthy operation:
```bash
# API logs - should show health check requests
docker-compose logs api 2>&1 | tail -20

# Worker logs - should show ready state
docker-compose logs worker 2>&1 | tail -20

# DB logs - should show connections
docker-compose logs db 2>&1 | tail -10

# Redis logs - should show ready
docker-compose logs redis 2>&1 | tail -10
```

### 7. Integration Test (Optional but recommended)
Run a quick smoke test:
```bash
# Register a test user (will fail with duplicate, but tests API connectivity)
curl -X POST http://localhost:8080/api/v1/auth/register \
  -H "Content-Type: application/json" \
  -d '{"email":"test@example.com","password":"testpass123"}'
```

### 8. Cleanup & Reporting
- Capture all logs as artifacts
- Stop services: `docker-compose down -v`
- Report status

## Artifact Collection (on failure or always)
```bash
mkdir -p logs/
docker-compose logs > logs/all-services.log
docker-compose logs api > logs/api.log
docker-compose logs worker > logs/worker.log
docker-compose logs db > logs/db.log
docker-compose logs redis > logs/redis.log
docker-compose logs nginx > logs/nginx.log
docker-compose ps > logs/container-status.txt
```

## Implementation File
`.github/workflows/docker-compose-test.yml`

## Key Considerations
1. Worker health: No healthcheck in docker-compose, must verify via logs
2. Nginx: No healthcheck defined, verify via successful proxy to API
3. Log patterns: Need to check actual worker logs to determine "healthy" patterns
4. Network: Services use ytprocessor-network
5. Volumes: postgres_data and storage volumes need to be cleaned up

## Success Criteria
- [ ] All 5 services start successfully
- [ ] api, db, redis report Docker health status = "healthy"
- [ ] Direct health endpoint (localhost:8000/api/v1/health) returns OK
- [ ] Proxied health endpoint (localhost:8080/api/v1/health) returns OK
- [ ] Worker logs show successful startup (no errors, connected to Redis)
- [ ] No ERROR/FATAL messages in any service logs during startup
